# MeetAgent 可信会议 Agent 阶段 1 实施计划

## 目标

建立最小可信闭环：Agent 回答仍保持现有体验，但工具结果开始携带结构化来源，最终回答可以在前端展示“引用来源”。

阶段 1 不做 verifier、不做 reranker、不强制每句话都带 `[S1]`。先保证证据链路打通。

## 范围

本阶段实现：

1. 新增统一数据结构 `Source` / `ToolResult`。
2. 新增兼容式 `execute_tool_structured()`。
3. 保留旧 `execute_tool()`，避免影响后台任务和已有调用。
4. 优先结构化以下工具：
   - `search_meetings`
   - `get_action_items`
   - `get_decisions`
   - `get_risks`
   - `list_meetings`
5. Agent loop 汇总工具 sources。
6. `/agent/qa` 和 `/agent/qa/stream` 返回 sources。
7. 前端在回答下方展示引用来源。
8. 历史会话从 `tool_calls_log` 中恢复 sources 展示。

## 不做内容

- 不引入 verifier / 重写 Agent。
- 不引入 reranker。
- 不修改数据库 schema。
- 不强制 LLM 每个句子输出 `[S1]`。
- 不把 `note_id`、`chunk_id` 直接展示给普通用户。

## 数据结构

`Source` 用于表示可展示的证据来源：

```text
source_id      前端和回答中展示的引用编号，如 S1
note_id        内部会议 ID，仅后端使用
chunk_id       内部片段 ID，仅后端使用
meeting_title  会议标题
create_time    会议时间
speaker        发言人
quote          原文或结构化记录内容
score          可选分数
```

`ToolResult` 用于统一工具返回：

```text
ok            工具是否成功
tool          工具名
text_for_llm  给模型看的文本
data          给系统使用的结构化数据
sources       证据来源列表
error         错误信息
```

## 实施步骤

1. 新增 `app/agent/types.py`。
2. 改造 `app/agent/tools.py`：
   - 新增结构化工具实现。
   - `execute_tool_structured()` 返回 `ToolResult`。
   - `execute_tool()` 继续返回 `text_for_llm`。
3. 改造 `app/agent/loop.py`：
   - 使用结构化工具执行。
   - 汇总并重新编号 sources。
   - `run_agent()` 和 `run_agent_stream()` 返回 sources。
4. 改造 `app/main.py`：
   - Agent 响应模型增加 `sources`。
   - SSE done 事件携带 sources。
5. 改造 `app/agent/session.py`：
   - 从历史 `tool_calls_log` 中恢复 assistant 消息的 sources。
6. 改造前端：
   - 当前流式回答完成后记录 sources。
   - 历史消息加载 sources。
   - `ChatWindow` 在回答下方展示来源。

## 验收标准

1. 用户问会议相关问题时，回答下方出现“引用来源”。
2. 引用来源显示会议标题、日期、发言人和摘录。
3. 前端不显示 `note_id`、`chunk_id`、`user_id`。
4. 原有工具调用、流式回答、多会话后台运行不被破坏。
5. `python -m compileall app` 和 `npm run build` 通过。


一、综合检索类
  这些适合测试 search_meetings、get_topic_history、摘要综合能力。

  1. 最近几次会议里，关于“孩子手机管理”的观点有哪些？请按不同观点归纳。
  2. 关于“孩子学习动力不足”，会议里提到的原因有哪些？哪些是家长造成的，哪些是学校或孩子自身因素？
  3. 请总结一下这个用户最近会议中反复出现的家庭教育核心理念。
  4. 最近会议中多次提到“价值感”，它具体指什么？有哪些提升孩子价值感的方法？
  5. 关于“夫妻关系优先于亲子关系”，会议里有哪些论据和案例？
  6. 会议中提到的“先搞人，再搞事”是什么意思？它和处理学习、手机、师生关系有什么关系？

  二、冲突/不同观点测试
  这些最适合测试 Agent 是否会混乱，因为会议里存在看起来有张力的观点。

  1. 会议里一方面说要放手、少管孩子，另一方面又说要制定手机规则，这两种观点是否冲突？应该如何理解？
  2. 有的会议说“教育孩子可以适度打骂但要讲方法”，也有很多内容强调认同、接纳、避免批评指责。这里是否存在矛盾？请结合会议内容解释。
  3. 关于孩子手机问题，会议里有没有出现“不能完全禁止”和“需要规则管理”两种不同思路？请对比它们。
  4. 会议里既强调孩子要自由支配时间、玩耍和同伴互动，又强调要自动自发学习、追求目标。这两个方向怎么平衡？
  5. 会议里既说“不能只盯成绩”，又说要培养真正的学霸和提升学习能力，这是否冲突？
  6. 关于课程成交，会议里既说教育要纯粹为孩子好，又有大量报名、成交、目标冲刺安排。请分析这两类内容之间的关系和潜在冲突。
  7. 亲子陪伴营复盘中，安全第一、拍照宣传、成交转化这几个目标之间有没有冲突？会议中提出了哪些解决办法？
  8. 会议中既提到公益、免费赠送课程，也提到收费能提升学员意愿，这两个观点如何统一？

  三、决策/待办/风险测试
  这些适合测试结构化工具是否能正确调用。

  1. 这个用户最近有哪些待办事项？请按负责人分组。
  2. 12月团队冲刺目标相关的待办、决策和风险分别是什么？
  3. 关于讲师外部授课和成交报备，会议中做了哪些决策？有哪些风险？
  4. 关于报销规范，会议中有哪些明确要求？如果不执行会有什么风险？
  5. 关于明年直播和新媒体战略，会议中做了哪些决策和后续动作？
  6. 亲子陪伴营复盘中有哪些需要改进的待办？哪些和安全有关？
  7. 最近会议中风险最高的事项有哪些？请按“家庭教育风险”和“业务运营风险”分类。
  8. 请找出所有和“安全第一”相关的风险、决策和待办。

  四、跨会议主题追踪
  这些适合测试主题历史能力。

  1. 请追踪“成长架构”这个主题在不同会议中的讨论变化。
  2. 请梳理“手机游戏/手机管理”在不同会议中的观点演变。
  3. 请追踪“情绪稳定”这个主题，它在哪些会议中被提到，分别解决什么问题？
  4. 请追踪“师生关系”相关内容，会议里有哪些案例、风险和解决建议？
  5. 请追踪“家长自我成长”这个主题，会议中有哪些反复强调的观点？
  6. 请追踪“成交980课程/成长架构报名”相关内容，会议中的策略、风险和行动有哪些？

  五、容易暴露系统问题的问题
  这些问题可以看 Agent 是否会编造、是否会泄露内部 ID、是否会把不同会议混成一个结论。

  1. 请列出最近会议中所有明确提到的金额、目标和处罚，并说明分别来自什么场景。
  2. 哪些会议提到了“980元”“25万”“260万”“5000元罚款”？这些数字分别代表什么？
  3. 请比较“亲子陪伴营活动部署”和“亲子陪伴营复盘”中关于安全管理的差异。
  4. 请找出会议中关于“孩子辍学/休学/厌学”的不同原因解释，并说明哪些是案例，哪些是讲师观点。
  5. 会议中有没有提到可能误诊多动症、非法行医或迷信风险？这些风险分别对应什么背景？
  6. 请判断：会议是否认为孩子问题主要来自孩子自己？如果不是，会议中给出的主要责任链条是什么？
